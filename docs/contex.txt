  [HIGH] CF-1: DB write na każde zamówienie bez listu przewozowego — Neon nigdy nie śpi                                                                                                                  

  Symptom: Każde zamówienie w statusie shipped/processing+SENT, któremu Allegro jeszcze nie przypisało listu przewozowego, generuje zapis do DB co ~25 minut.

  Wpływ biznesowy: Neon pozostaje aktywny przez całą dobę bez realnego ruchu, generując koszty compute i rachitując ideę KV idle guard.

  Przyczyna źródłowa: tracking-refresh.ts:227-238 — gdy Allegro zwróci 0 przesyłek dla zamówienia, funkcja wykonuje bezwarunkowy db.update(orders).set({ trackingStatusUpdatedAt: now }). Cooldown 25
  minut zapobiega zbyt częstym zapisom, ale nie zeruje aktywności — zamówienie jest zawsze kandydatem do odświeżania dopóki nie pojawi się list przewozowy.

  // tracking-refresh.ts:227-238
  if (shipmentCandidates.length === 0) {
    await db.update(orders)
      .set({ trackingStatusUpdatedAt: now, updatedAt: now })
      .where(and(
        eq(orders.id, orderId),
        sql`${orders.trackingStatusUpdatedAt} IS NULL OR ...`
      ))
    return buildRefreshOutcome(before, before)
  }

  Reprodukcja: Opłać zamówienie Allegro. Zanim kurier przypisze numer śledzenia (minuty–godziny), każdy cron trackingowy uderza w Allegro /shipments, nie dostaje nic, i zapisuje timestamp do DB. Neon
  nie śpi.

  [HIGH] CF-4: Przejście processing → delivered nigdy nie zachodzi automatycznie

  Symptom: Jeśli admin nie oznaczy zamówienia jako shipped przed tym jak kurier dostarczy paczkę, zamówienie zostaje w processing na zawsze — mimo że tracking potwierdza dostarczenie.

  Wpływ biznesowy: Admin widzi zamówienie jako otwarte (processing), klient może składać reklamacje "gdzie jest paczka", wysyłane są niepotrzebne powiadomienia.

  Przyczyna źródłowa: tracking-refresh.ts:332-344 — filtr eq(orders.status, 'shipped') blokuje przejście dla zamówień, które nigdy nie przeszły przez status shipped:

  // tracking-refresh.ts:332-344
  await db.update(orders)
    .set({ status: 'delivered', deliveredAt: ... })
    .where(and(
      eq(orders.id, orderId),
      eq(orders.status, 'shipped'),   // ← blokuje processing → delivered
    ))

  Reprodukcja: Zamówienie z Allegro → status processing → kurier dostarcza bez przejścia przez shipped w adminie → tracking zwraca DELIVERED → status nie zmienia się.

  [HIGH] CF-NO-CANDIDATES: Zamówienia paid+SENT są wykluczone z kandydatów do trackingu

  Symptom: System raportuje "brak kandydatów" mimo istnienia nowych przesyłek. To jest znany bug.

  Przyczyna źródłowa: selectTrackingRefreshCandidates (tracking-refresh.ts) uwzględnia tylko:
  - status IN ('shipped', 'delivered')
  - status = 'processing' AND allegroFulfillmentStatus IN ('SENT', 'PICKED_UP')

  Wyklucza: status = 'paid' z allegroFulfillmentStatus = 'SENT' — co zdarza się gdy event BOUGHT jest procesowany PO evencie fulfillmentu (race condition), albo gdy reconcileOrder nie zdążył przejść
  statusu z paid do processing przed wywołaniem trackingu.

  // Pole where w selectTrackingRefreshCandidates — brakuje:
  // status='paid' AND allegroFulfillmentStatus IN ('SENT','PICKED_UP')
  // status='pending' AND allegroFulfillmentStatus IN ('SENT','PICKED_UP')

  Weryfikacja: Uruchom na bazie:
  SELECT id, status, allegro_fulfillment_status
  FROM orders
  WHERE source='allegro'
    AND allegro_fulfillment_status IN ('SENT','PICKED_UP')
    AND status NOT IN ('shipped','delivered','processing');

  Jeśli wynik > 0 — bug jest aktywny.

  [MEDIUM] AVC-4: OrderDetailModal wyświetla timeline oparty o stary status z listy, nie o odświeżony tracking

  Symptom: Po auto-odświeżeniu trackingu w modalu, timeline zamówienia nadal renderuje stary stan.

  Przyczyna źródłowa: OrderDetailModal.tsx:256-260 — resolveShipmentStatus zawsze czyta order.status (z payload listy, nie z odpowiedzi tracking refresh):

  // OrderDetailModal.tsx:256-260
  const resolvedShipmentStatus = resolveShipmentStatus({

  Obiekt tracking (linia 154) zawiera status, ale nigdy nie jest użyty w resolveShipmentStatus. Modal wyświetla poprawny shipmentDisplayStatus z refreshu, ale błędny krok w timeline.

  2. Luki logiczne i nonsensowne reguły biznesowe

  ---
  LG-1: Zamówienia delivered są odświeżane przez 30 dni po dostarczeniu

  Kandydaci do trackingu z HARD_CUTOFF_DAYS = 30 uwzględniają dostarczone zamówienia co 12 godzin przez miesiąc. Dla typowego wolumenu (kilkadziesiąt zamówień/miesiąc) to ~2 zapytania/dobę na
  zamówienie × 30 dni × N zamówień. Celowe czy przeoczenie — powinno być udokumentowane.

  LG-2: runTrackingBackfillPage nie ma cooldownu

  Wywołana wielokrotnie z adminu odpytuje Allegro carrier API za wszystkie 180 dni zamówień bez żadnego deduplication. Brak blokady KV — równoległe wywołania zduplikują subrequesty.

  LG-3: Podwójne odpytanie /shipments — sync.ts i tracking-refresh.ts

  syncMissingTrackingNumbers w sync.ts i refreshOrderTrackingSnapshot w tracking-refresh.ts niezależnie uderzają w GET /order/checkout-forms/{id}/shipments dla tych samych zamówień. Brak konfliktu
  danych (piszą różne kolumny), ale podwójny koszt subrequestów dla zamówień czekających na numer śledzenia.

  LG-4: reconcileOrder czyta waybill z ciała checkout-form, nie z /shipments

  handlers.ts:491 — form.delivery?.shipmentSummary?.waybill jest obecne w odpowiedzi Allegro tylko gdy zamówienie ma dane dostawy. Jeśli nieobecne — trackingNumber pozostaje null i jest backfillowany
  przez osobny job. Jeśli ten job nie działa (KV idle guard = '0') — zamówienie może nie mieć numeru śledzenia przez wiele godzin.

  6. Brakujące testy

  1. paid+SENT → musi być kandydatem do trackingu (obecnie wykluczone)
  2. processing+DELIVERED → auto-przejście do delivered (CF-4 blokuje)
  3. Multi-package: shipment A=EXCEPTION, shipment B=DELIVERED → system nie powinien cicho pomijać wyjątku
  4. Modal: po auto-refresh trackingu timeline używa nowego status, nie starego z listy (AVC-4)
  5. Zamówienie bez listu przewozowego: cron nie powinien generować DB write przy każdym ticku
  6. subrequestLimitHit guard: zmiana wiadomości błędu CF nie powinna wyciszyć ochrony
  7. Backfill: równoległe wywołania runTrackingBackfillPage nie duplikują subrequestów
  8. HARD_CUTOFF_DAYS: zamówienie shipped 31 dni temu nie pojawia się jako kandydat
